# 安全模型

本文档定义了 2FA MVP 的安全边界。任何违反这些规则的实现都必须在
评审中被拒绝。

## 1. 两个独立密钥，两项独立职责

系统中存在两个独立密钥，由不同方持有，用于不同目的。

### 1.1 账号密码（服务端可知，服务端可验证）

- **用途：** 向同步服务认证用户身份。授权对该用户密文密钥库的读写操作。
- **生命周期：** 仅在注册、登录和修改密码时通过 TLS 发送至服务端。
  在服务端仅以加盐哈希形式存储，使用内存密集型算法（推荐 Argon2id 作为默认值；
  实现者需记录并固定参数）。
- **服务端能力：** 将提交的密码与哈希值比对。服务端**无法**恢复明文。
- **恢复策略：** **MVP 不提供账号密码恢复功能。** 忘记账号密码意味着用户
  必须重新注册；即使用户通过某种方式重新获得账号访问权限，服务端也无法
  恢复密文（因为服务端无法解密任何内容）。

### 1.2 同步密码（仅客户端持有，永不发送）

- **用途：** 派生加密密钥，用于保护 TOTP secret、展示字段以及密文负载中的
  分组展示数据。
- **生命周期：**
  - 仅在扩展中由用户输入。
  - **绝对不得**出现在 HTTP 请求体、URL、请求头、Cookie、查询字符串、
    日志行或遥测信息中。
  - 密钥库解锁期间可在扩展内存中暂存。锁定/退出/超时后必须清除。
  - 可通过便利优先的本地持久化方案间接持久化（参见第 3 节）；绝不以明文形式持久化。
- **服务端能力：** 无。服务端不存在任何处理同步密码的路径。
- **恢复策略：** **不提供服务端同步密码恢复。** 丢失同步密码且没有其他设备
  持有该密码，意味着密文无法恢复。UI 必须在设置同步密码时清楚说明此后果。

### 1.3 为什么要两个密码

这种分离保证了：

1. 修改账号密码**不会**要求重新加密所有用户数据。
2. 修改同步密码**不会**要求撤销服务端会话。
3. 即使攻击者恢复了密码哈希和所有已存储的会话，服务端被攻破也不会
   导致明文泄露。

## 2. 仅客户端密钥的信封加密

扩展使用**信封加密**：

```
同步密码 ──KDF──▶ KEK_sync（密钥加密密钥，按用户派生）
                           │
                           ▼
                  包裹 DEK_vault（随机数据加密密钥）
                           │
                           ▼
         DEK_vault 加密每条记录体（每条记录一个随机 nonce）
```

具体而言：

- `KEK_sync` 由同步密码通过内存密集型 KDF（推荐：Argon2id）派生，
  并绑定到服务端以非密钥形式存储在每个用户的 `kdf_salt` 上。
- `DEK_vault` 是首台设备生成的随机 256 位密钥，用 `KEK_sync` 包裹后
  作为 `wrapped_dek` 上传。其他设备获取 `wrapped_dek`，使用从相同同步密码
  派生的同一 `KEK_sync` 解包。
- 每条记录体（条目、分组展示 blob）使用 `DEK_vault` 通过 AEAD 加密
  （推荐：AES-256-GCM 或 XChaCha20-Poly1305），每条消息使用独立的随机 nonce。
- 传输格式为 `EnvelopeV1`（参见 `docs/openapi.yaml`）：
  密钥库信封使用
  `{ alg, kdf, kdf_params, kdf_salt_b64, wrapped_dek_b64, wrap_iv_b64 }`，
  记录体使用
  `{ alg, iv_b64, ct_b64, aad_b64? }`。

具体算法名称、KDF 参数和版本常量由扩展实现者选定并在代码中固定。
契约通过携带 `alg`/`kdf` 字段支持未来的版本变更。

### 2.1 同步密码变更

1. 客户端使用当前 `KEK_sync` 解包 `DEK_vault`。
2. 客户端从新同步密码派生新的 `KEK_sync'`。
3. 客户端用 `KEK_sync'` 包裹 `DEK_vault`，生成新信封。
4. 客户端上传新信封至同一条记录（`vault.envelope`）。
5. 其他设备在下一次同步时，提示用户输入新的同步密码以解包相同的 `DEK_vault`。
6. 已存在的记录密文**不会**被重新加密，因为 `DEK_vault` 未发生变化。
   这是有意为之，也是密钥轮换成本低廉的设计原因。

### 2.2 忘记同步密码

不存在恢复路径。UX 必须在用户接受同步密码*之前*明确说明此后果。

## 3. 便利优先的本地持久化

MVP 默认采用便利优先的本地持久化方案，使得已解锁的扩展可以重新向服务端认证
并重新派生密钥，而无需每次会话都让用户重新输入同步密码。

### 3.1 机制（Chromium MV3）

1. 首次解锁时，扩展通过
   `crypto.subtle.generateKey({ name: "AES-GCM", length: 256 }, /* extractable */ false, ["encrypt", "decrypt"])`
   生成一个**不可导出**的 AES-GCM `CryptoKey`。
2. 该 `CryptoKey` 以 `CryptoKey` 引用形式存储在 **IndexedDB** 中。
   因为它不可导出，无法通过 `crypto.subtle.exportKey` 导出原始字节。
3. 同步密码（或派生的 `KEK_sync`）用该包裹密钥加密，密文存储在
   `chrome.storage.local` 中。
4. 下一次会话时，扩展读取密文，使用 IndexedDB 中的 `CryptoKey` 原地解密。

### 3.2 此方案能防护什么

- 仅凭扩展的 `chrome.storage.local` 目录的简单复制不足以恢复同步密码，
  因为密文在没有 IndexedDB 中的包裹密钥时毫无用处。
- 仅对两个存储之一进行只读转储也是不够的。

### 3.3 此方案不能防护什么

- **已被攻破的浏览器配置文件。** 能在用户 Chromium 配置文件中执行代码的
  攻击者可以要求同一个 `CryptoKey` 解密相同的密文。"不可导出"意味着
  "字节不会离开浏览器"，但并不意味着"同一配置文件上的攻击者无法使用该密钥"。
- 在同一浏览器中以正确权限运行的恶意扩展，或任何浏览器级别的恶意软件，
  原则上可以访问相同的 API。
- 对同时获取的两个存储加内存中密钥材料的磁盘取证恢复。

### 3.4 用户控制

- 用户可以选择退出便利持久化，要求每次浏览器会话都输入同步密码
  （`requireSyncPasswordEachSession`）。
- 退出远程账号时，用户可以选择清除本地持久化的同步凭证。
- 自动锁定计时器会从扩展内存中清除已解密的明文，但可能保留已包裹的同步凭证
  以供重新解锁。

设置 UI 必须用通俗语言说明这些权衡（参考：需求文档第 3.2 和 5.5 节）。

## 4. 服务端保障

服务端必须：

- 使用内存密集型 KDF（Argon2id）对账号密码进行哈希。
- 在生产部署中对所有外部流量使用 TLS。
- 拒绝携带可疑请求头或类似于同步密码的参数名的请求（纵深防御）。
- 对 `/v1/auth/*` 和 `/v1/sync/*` 实施速率限制，各自独立配额
  （参见 `.env.example`）。
- 绝不记录同步端点的请求体。
- 绝不记录 `Authorization` 请求头、Cookie 或任何*看起来像*秘密材料的字段
  （任何名为 `password`、`secret`、`wrapped_dek`、`ct_b64`、`iv_b64`、
  `kdf_salt_b64` 的字段）。
- 对不存在的用户的登录请求，使用与密码错误登录相同的时间/错误消息特征来拒绝。

服务端不得：

- 存储同步密码。
- 解密用户密文。
- 提供任何同步密码恢复功能。
- 允许管理员通过 `/v1/admin/*` 读取密文正文。

## 5. 会话与设备

- 登录后，服务端颁发绑定到设备记录（由客户端选择的随机 `device_id`，
  因此服务端不会看到硬件标识符）的不透明会话令牌。
- 用户可以列出自己的设备并撤销任意设备。撤销会在下一次请求时立即使会话失效。
- 账号密码变更可由实现者决定是否使其他会话失效（推荐这样做）。
- 管理员可以通过 `/v1/admin/*` 撤销任意设备或会话。撤销操作不影响存储中的密文。

## 6. 威胁模型概要

- **对手 A（网络）：** 只能以密文形式读取 TLS 保护的线字节。AAD 防止对信封参数的篡改。
- **对手 B（服务端运维/数据库管理员）：** 可见密文、KDF 盐值、已包裹的 DEK、
  账号哈希、同步元数据（rev、seq、大小、时间戳、分组 ID）。无法推导出条目明文、
  条目名称、分组展示名称、secret、恢复信息、备注。
- **对手 C（已被攻破的浏览器配置文件）：** 胜出。便利优先的本地持久化明确
  排除在此对手范围之外。已告知用户此风险。
- **对手 D（丢失/被盗的未锁定设备）：** 自动锁定和手动锁定是缓解措施。
  锁定后，已包裹的同步凭证能够抵抗离线攻击的程度等同于
  带有随机不可导出密钥的 AES-256-GCM 所能抵抗的程度。

## 7. 验收锚点

`requirements.md` 中以下条目对应安全要求：

- AC4：网络抓包和服务端日志不得暴露同步密码、`DEK_vault` 或 TOTP secret。
- AC5：拥有原始数据库访问权限的数据库管理员只能看到条目和分组展示字段的密文。
- AC7：错误的同步密码必须表面解密错误，且不得覆盖本地或远程数据
  （参见 `sync-protocol.md`）。
- AC9：管理操作的范围限定于非敏感元数据。
